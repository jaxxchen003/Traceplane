# Discussion Insights

这份文档从 [原始讨论记录.md](/Users/jaxxchen/projects/enterprise-agent-work-graph/原始讨论记录.md) 中提炼项目真正有效的思路演进，用于服务当前产品判断。

它不是 PRD，也不是最终对外表达，而是“为什么我们现在会把项目定义成这样”的压缩版记录。

## 1. 项目思路的演进路径

### 阶段一：AI 时代的云存储会变成什么
最初问题不是企业管理，而是一个更宽的命题：

- 传统网盘是“人上传、人检索、人分享”
- Agent 时代需要“Agent 可调用、可理解、可写入”的数据基础设施
- 存储对象从文件扩展到 `JSON`、`Markdown`、脚本、日志、`HTML/SVG`、监控数据、上下文片段

这一阶段的核心贡献是把“Agent 数据”从文件概念中解放出来。

### 阶段二：这不是网盘，而是 Data OS
随后讨论把产品理解推进到“Personal / Enterprise Data OS”：

- 底层不只是文件存储，而是多种数据类型的统一数据节点
- 上层要同时服务人和 Agent
- 系统需要原生考虑加密、权限和审计
- MCP 被看作 Agent 的标准接入协议

这一阶段的核心贡献是：产品不再是“更好的云盘”，而是“Agent 可操作的数据基础设施”。

### 阶段三：从个人数据节点升级到企业数据中枢
最后一次关键跃迁是从“Agent 的持久记忆”转向“企业如何管理一组 Agent”：

- 记忆只是输入，不是完整产品
- 企业管理者需要看见输入、过程、输出的闭环
- 多 Agent 协作不仅要共享记忆，还要共享日志、文件和规则
- 真正可采购的能力是治理、权限、审计和可复盘

这一阶段直接导向当前定义：
`Enterprise Agent Work Graph`

## 2. 从原始讨论沉淀出的核心判断

### 2.1 数据对象已经变了
Agent 时代的核心对象不只是文件，而是三类工作资产：

- 输入型资产：Memory / Context
- 过程型资产：Trace / Log / Decision
- 输出型资产：Artifact / Report / Script / HTML / SVG / JSON

这正是当前 `memory + trace + artifact` 三分法的来源。

### 2.2 真正稀缺的是管理闭环
原始讨论里最重要的一条判断是：

企业不只需要 Agent “记住”，更需要 Agent “可管理”。

也就是：

- 知道它读了什么
- 知道它做了什么
- 知道它产出了什么
- 知道它依据什么规则行动
- 出问题时能审计和追责

### 2.3 共享不等于同一个 Workspace
一个关键认知转折是：

多 Agent 协作场景下，不应该简单把所有 Agent 放进同一个工作区。

更合理的是：
- 在同一数据平面上受控共享
- 通过权限、规则和审计来决定哪些节点和关系能被访问

这直接支撑了后续的 `shared_with`、`restricted_by`、`access_grants`、`audit_events` 设计。

### 2.4 Memory 应该降级成模块，不该等于产品
原始讨论里很明确地指出：

- 现有 memory layer 玩家主要补“输入侧”
- 你的方向更像“Agent 工作资产中心”
- 所以 memory 应该是系统中的一个模块，而不是产品本身

这也是当前产品边界能成立的前提。

## 3. 对当前项目最重要的战略启发

### 3.1 我们不是在做这些东西
- 不是企业网盘
- 不是单点 Agent memory API
- 不是通用 workspace
- 不是给人类优先编辑文档的工具

### 3.2 我们真正要做的是
一个面向企业多 Agent 协作场景的共享数据平面和控制平面：

- 数据平面承载 Memory、Trace、Artifact
- 控制平面承载 Policy、Permission、Audit
- `Episode` 作为工作主线，把输入、过程、输出连成图

### 3.3 差异化不在“能不能存”
原始讨论有个很重要的隐含结论：

云厂商、网盘厂商和现有工作台都能存数据，但它们通常不同时具备：

- Agent 原生写入和读取
- 语义级和任务级组织方式
- 跨 Agent 受控共享
- 历史版本和来源回链
- 企业级治理、审批、审计

所以差异化不在 storage，而在“工作图谱 + 治理闭环”。

## 4. 原始讨论里仍然值得保留的早期思想

虽然当前方向已经偏企业场景，但早期讨论里有几条仍然值得保留。

### 4.1 Developer-first 很重要
早期讨论强调：

- `CLI + MCP` 比先做重 UI 更适合早期验证
- Agent 和开发者需要的是低摩擦写入、读取和调试能力

这意味着即使未来做 manager 视图，底层产品也不该脱离开发者可用性。

### 4.2 数据类型分层是对的
原始讨论把数据分成：

- 人类产物
- 机器产物
- 实时流数据

这个分法本身仍有价值，说明后续 schema 和存储后端不该假设所有数据都走同一路径。

### 4.3 安全不应该后补
早期就反复出现的关键词包括：

- 加密
- 身份
- 权限
- 审计
- 不可篡改日志

即使 MVP 不做全量零知识架构，这些能力也应该从 schema 和事件模型上预留。

## 5. 当前项目与原始讨论相比，已经明确收敛的地方

下面这些问题，在当前项目里已经比原始讨论更清楚了：

- 从“个人数据节点”收敛到“企业级 Agent 工作图谱”
- 从“泛化云端存储”收敛到 `memory / trace / artifact`
- 从“宏大基础设施叙事”收敛到 `Episode` 主线
- 从“广义安全”收敛到 `policy / permission / audit`

这意味着接下来不应再回到“什么都能存、什么都能管”的泛平台路线。

## 6. 仍然没有完全定下来的问题

原始讨论还留下几类未完全定稿的问题：

- 到底先偏 developer-first，还是先偏 manager-first
- MCP / CLI / HTTP API 三种入口的先后顺序
- Node 级授权是否进入 MVP
- 加密与性能的平衡如何处理
- 对 Obsidian / Notion 的关系是集成入口还是竞品替代

这些问题已经被转移到 [open-questions.md](/Users/jaxxchen/projects/enterprise-agent-work-graph/docs/open-questions.md) 里继续收敛。

## 7. 结论

原始讨论最有价值的，不是它给出的所有答案，而是它清楚展示了这个项目的方向是如何逐步收敛出来的：

1. 从“AI 时代的新存储形态”出发
2. 识别出 Agent 数据不只是文件，而是上下文、过程、产物
3. 进一步意识到企业真正购买的是管理闭环，而不是记忆本身
4. 最终收敛到 `Enterprise Agent Work Graph`

一句话总结：

这个项目不是在做“让 Agent 不失忆”，而是在做“让企业能管理 Agent 的工作链路”。
