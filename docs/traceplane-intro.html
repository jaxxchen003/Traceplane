什么是 Traceplane
Multi-Agent Continuity Layer — 让 AI Agent 的工作永不断档

Traceplane 是一个为多 AI Agent 协作场景设计的连续性层（Continuity Layer）。它不是让单个 Agent 变得更聪明，而是解决多个 Agent 在同一任务上协作时的工作断档问题。

当你同时使用 Claude Code、OpenCode、Gemini CLI 等 AI Agent 时，每次切换 Agent 都意味着重新解释上下文——这是巨大的浪费。Traceplane 让这些 Agent 共享一个可观测、可交接、可恢复的统一工作线程。

为什么存在
核心痛点：多 Agent 协作场景下的上下文断裂

Traceplane 解决的不是单个 Agent 的能力问题，而是多 Agent 协同工作时的结构性痛点。以下三个场景在日常 AI 辅助开发中高频出现：

# 痛点场景 具体表现 损失
01 上下文重复传递 同样的背景、目标、约束反复讲给不同的 Agent ⚡ 浪费大量 Token 和时间
02 工作中途断档 Agent A 做到一半，Agent B 无法知道进展和未完成任务 ⚡ 工作无法继续，重复劳动
03 成果散落无法复盘 工作分散在多个聊天窗口和会话记录中 ⚡ 无法审计、无法学习、无法复用

这三个痛点的本质是同一个问题：AI Agent 缺乏一个持久化、结构化的工作状态载体。Traceplane 就是这个载体。

核心概念
四个关键抽象，构建 Agent 连续性工作体系
🧠 Episode 意识载体
▼

Episode 是 Traceplane 最核心的抽象。它不是日志行，而是 Agent 的完整执行上下文——包含目标、每一步的工具调用与结果、中间产物、风险标记、最终产出。


一个 Episode 可以被恢复（继续未完成任务）、复用（相同目标的新执行）、交接（移交给另一个 Agent 继续）。这使得 Agent 切换不再意味着上下文清空。

🔀 Surgical Replay 时间分叉
▼

Surgical Replay 让你像 git checkout 一样，从 Episode 历史中的任意节点 Fork 出一条新的执行路径。


当 Agent 走了一条错误路径时，不需要从头开始——直接回到分叉点，以不同的方式继续。这是对 AI 工作流的版本控制能力。

📋 Handoff Brief 自动交接
▼

Handoff Brief 是 Traceplane 自动从 Episode 数据中生成的结构化交接文档，包含四个部分：


• 目标 — 这个 Episode 要完成什么
• 进展 — 已经完成了什么，走到了哪一步
• 产物 — 生成了哪些文件、数据、结果
• 风险 — 遇到了哪些问题，有哪些注意事项


下一个 Agent 收到 Handoff Brief 后可以直接继续，无需人工解释。

👁️ 全栈可观测性 统一视角
▼

Traceplane 将来自多个 Agent 的行为数据统一汇聚到同一个观察面板：Agent 执行的每个工具调用、生成的产物、检测到的风险行为、关键指标——全部在同一个台面上可见。


这不只是日志，是结构化的可观测性层，支持审计、复盘和优化 Agent 工作流。

架构设计原则

Traceplane 采用「开发零门槛、生产高可用」的双轨策略：本地开发用 SQLite 一键启动，生产部署切换到 Supabase + R2 获得企业级可靠性。MCP 协议作为一等公民，确保与主流 AI Agent 的原生集成无需额外适配层。

接入方式
三种接入路径，适配不同使用场景

01 Agent SDK

适用于在代码中直接集成 Traceplane，完整的 TypeScript 类型支持。

TypeScript
import { TraceplaneSDK } from '@traceplane/agent-sdk';

const sdk = new TraceplaneSDK({
  baseUrl: 'https://traceplane.cc',
  projectId: 'your-project-id',
  agentId: 'claude-code',
});

// 创建 Episode
const session = await sdk.startEpisode({
  title: '修复登录 Bug',
  goal: '找到并修复用户无法登录的问题',
});

// 追加执行步骤
await session.step('分析错误日志', '发现 JWT 过期处理逻辑缺失');
await session.step('修改 auth.ts', '添加 token refresh 逻辑');

// 完成 Episode
await session.complete('Bug 已修复，已通过测试');

02 REST API

适用于任何语言、任何环境，通过标准 HTTP 接口接入。

Shell
# 创建 Episode
$ curl -X POST https://traceplane.cc/api/episodes \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"title":"修复登录Bug","goal":"找到并修复问题","agentId":"claude-code"}'

# 追加 Trace
$ curl -X POST https://traceplane.cc/api/episodes/{id}/traces \
  -d '{"step":"分析日志","result":"发现JWT问题"}'

为什么是 CLI
Agent 在 Terminal 里工作，不在浏览器里

当前 Traceplane 已有 Agent SDK（代码级集成）和 MCP 工具（协议级集成），但缺少一个关键的体验层：CLI 命令行体验层。

🖥️
终端原生
AI Agent（Claude Code、Gemini CLI）的工作场所是 Terminal，不是浏览器或 IDE 面板
⚡
零摩擦接入
无需修改代码，一行命令即可开始追踪，对现有工作流零侵入
🔄
守护进程
watch 模式可在后台静默运行，自动追踪 Agent 的每一次工具调用

命令设计
8 个核心命令，覆盖完整 Episode 生命周期
$ traceplane init 初始化当前目录，创建 .traceplane.json 配置文件
$ traceplane start "task" 创建新 Episode，指定任务目标描述
$ traceplane step "did X" 向当前 Episode 追加一条执行记录
$ traceplane complete 标记当前 Episode 为完成状态
$ traceplane brief 生成 Handoff Brief 并在终端格式化输出
$ traceplane watch 启动守护进程，自动监听并追踪 Agent 工具调用
$ traceplane fork <id> 从指定 Episode 节点 Fork 出新的执行路径
$ traceplane status 查看当前 Episode 状态、进度和最近记录
Watch 模式
守护进程实时监听，Agent 工具调用自动记录

Watch 模式是 CLI 的核心能力：以守护进程方式运行，监听 Agent 的标准输出流，自动解析工具调用事件并追加到当前 Episode，无需人工干预。

traceplane watch --agent claude-code
$ traceplane watch --agent claude-code
✓ [已启动] 监听中... Episode #42 进行中
[10:23:01] tool_use: Read(main.ts) → 已记录
[10:23:05] tool_use: Write(fix.ts) → 已记录
[10:23:09] tool_use: Bash(npm test) → 已记录
[10:23:12] ⚡ Handoff Brief 就绪 → 按 Ctrl+B 生成
$ 

工作原理

📡 输出流解析

监听 Agent 进程的 stdout/stderr，用正则/JSON 解析器识别工具调用事件，支持不同 Agent 的输出格式。

🔄 自动追踪

识别到 tool_use 事件后，自动调用 Traceplane API 的 append_trace 接口，将事件记录到当前活跃 Episode。

.claude/CLAUDE.md
## Traceplane 集成
## 在每次会话开始时自动追踪工具调用

在终端执行：traceplane watch --agent claude-code &

# 这会在后台启动 watch 守护进程
# 你的每一次工具调用都会被自动记录到 Traceplane
